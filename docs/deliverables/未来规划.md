# AutoSolver Agent 未来规划

## 1. 未来应该怎么做

项目后续应从“算法展示”升级为“业务场景驱动的即时履约调度 Agent”。决赛评审更关注创新性、完整性、应用效果和商业价值，因此下一阶段的目标应是让评委看到一个可运行、可解释、可落地的产品原型，而不是只看到算法日志。

推荐将产品定位明确为：

```text
美团即时履约调度 Agent：在高峰、雨天、骑手稀缺、低接单意愿等复杂场景下，自动生成更稳、更省、可解释、可回退的派单方案。
```

网页 Demo 应承担“产品演示”职责，PPT 承担“项目介绍”职责。网页中不需要重复介绍背景和架构，而应呈现 Agent 正在处理一个真实业务场景：它感知风险、生成方案、比较结果、解释决策，并输出业务指标。

建议优先补齐以下能力：

- 业务场景选择：将技术 case 改造成午高峰爆单、雨天低接单意愿、骑手稀缺商圈、合单机会密集等真实运营场景。
- 调度结果可视化：用城市网格、订单点、骑手点和派单连线展示最终调度结果。
- Before/After 对比：展示传统规则或贪心基线与 AutoSolver Agent 的指标差异。
- 决策解释面板：点击一个订单或任务组后，解释为什么选择这些骑手、为什么合单、为什么拒绝其他方案。
- 商业价值估算：用可调参数估算减少无人接单、降低履约成本、提升稳定性带来的经济收益。

## 2. 项目该如何优化

### 2.1 产品界面优化

当前网页更像算法运行面板，适合技术排查，但不适合决赛产品演示。建议改造成“即时履约调度指挥舱”。

页面第一屏应展示：

- 当前业务场景。
- 风险识别结果。
- 核心业务 KPI。
- 一键启动智能调度。
- AutoSolver Agent 相比基线的提升。

核心 KPI 建议包括：

- 预计完成率。
- 预计无人接单数。
- 履约成本指数。
- 骑手占用数。
- 相对贪心基线改善。
- 预计商业收益。

这些指标不一定都要直接来自官方评分，但必须在页面中标明是“本地仿真估算”或“业务估算指标”，避免把本地 cost 误包装成官方成绩。

### 2.2 Agent 展示优化

当前页面展示了 Perception、Planner、Executor、Critic、Controller、Memory、Evolution，但还偏技术。建议改成业务语言：

- 感知：识别订单量、骑手供给、接单意愿和合单机会。
- 规划：选择适合当前场景的调度策略。
- 执行：生成多个候选派单方案。
- 评估：比较方案质量，拒绝质量回退策略。
- 记忆：保留当前最优方案。
- 回退：实验策略失败时回到稳定方案。

页面应减少函数名和代码名的直出，把算法策略翻译成业务解释。例如：

- `single_multidispatch` 展示为“关键任务多骑手兜底”。
- `pair_matching` 展示为“合单机会匹配”。
- `production_solver` 展示为“综合调度优化器”。
- `evolution rollback` 展示为“实验策略未通过质量门，已安全回退”。

### 2.3 应用效果优化

决赛演示必须让评委直观看到项目效果。建议增加一个对比区：

```text
传统规则 / 贪心基线  vs  AutoSolver Agent
```

对比维度包括：

- 覆盖任务数。
- 预计完成率。
- 预计失败任务。
- 履约成本指数。
- 骑手使用数。
- 运行耗时。
- 方案稳定性。

这部分是应用效果的核心证据。即使算法内部很复杂，评委最终也需要看到“它比普通方案好在哪里”。

### 2.4 商业价值优化

建议在页面底部增加 ROI 模拟器，将算法效果转化为平台价值。

输入参数可以包括：

- 城市日订单量。
- 单个无人接单订单的损失。
- 履约成本指数折算金额。
- 部署商圈数量。
- 高峰时段订单占比。

输出可以包括：

- 预计每日减少损失。
- 预计每月节省履约成本。
- 预计减少无人接单订单数。
- 对用户体验、商家履约和骑手效率的影响。

这样可以把项目价值从“算法更优”转化成“平台能降本增效”。

## 3. 后续如何修改跟进

建议按三个阶段推进。

### 阶段一：决赛演示版

目标是在不大改算法的前提下，快速提升演示效果。

待办事项：

- 将首页改为“即时履约调度指挥舱”。
- 将样例名称改为业务场景名称。
- 增加业务 KPI 卡片。
- 增加 Baseline vs AutoSolver 对比。
- 增加任务级决策解释面板。
- 保留当前 Agent 事件流，但放到二级区域。

验收标准：

- 评委进入网页后，30 秒内能理解项目解决什么业务问题。
- 页面能展示 Agent 的工作过程，而不是只显示算法日志。
- 页面能看到 AutoSolver 相比基线的业务收益。

### 阶段二：产品原型版

目标是让 Demo 更接近真实运营工具。

待办事项：

- 增加城市网格或商圈示意图。
- 用订单点、骑手点和派单连线展示调度结果。
- 支持点击任务查看决策解释。
- 增加风险订单高亮，例如低接单概率、无人兜底、骑手稀缺。
- 增加多场景切换后的指标对比。

验收标准：

- 页面能像一个真实调度控制台，而不是展示网页。
- 非技术评委能理解 Agent 为什么这么派单。
- 产品价值可以通过指标和场景自然呈现。

### 阶段三：落地扩展版

目标是为后续真实业务接入预留方向。

待办事项：

- 接入更真实的订单、骑手、商圈、天气、时段特征。
- 将本地仿真指标与真实履约指标对齐。
- 增加策略 A/B 测试能力。
- 增加策略安全门和版本管理。
- 将自进化策略沉淀为可审计的策略库。
- 为运营人员提供策略解释和人工干预入口。

验收标准：

- 系统可以支持真实业务数据回放。
- 策略优化结果可以用业务指标衡量。
- Agent 的每次策略选择、接受、拒绝、回退都有审计记录。

## 4. 建议的决赛演示脚本

决赛现场建议按以下节奏演示：

1. 选择“雨天低接单意愿”或“骑手稀缺商圈”场景。
2. 展示 Agent 识别当前风险：接单意愿下降、骑手供给不足、合单组合复杂。
3. 点击“开始智能调度”。
4. 展示 Agent 自动生成多个候选方案，并拒绝质量回退策略。
5. 展示最终派单结果和 Baseline vs AutoSolver 指标提升。
6. 点击一个任务组，展示为什么这么派、为什么不选其他方案。
7. 展示 ROI 估算，说明平台降本、用户体验和商家履约价值。

最终目标是让评委形成一个清晰判断：

```text
AutoSolver Agent 不只是一个算法求解器，而是一个面向即时履约运营场景的智能调度产品原型。
```
