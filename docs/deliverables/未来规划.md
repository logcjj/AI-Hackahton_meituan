# AutoSolver Agent 未来规划

## 1. 目标定位

AutoSolver Agent 后续不应继续做成偏炫技的展示页，也不应只停留在算法日志页面。更合适的方向是把现有 Web demo 收敛成一个面向开发人员和技术评审的“调度分析工作台”：

```text
AutoSolver Agent 调度分析工作台：
在高峰、雨天低接单意愿、骑手稀缺、合单机会密集等场景下，
运行同一套 Agent 控制器，展示风险画像、策略选择、候选评估、
安全回退和最终派单结果。
```

这里的重点是“专业、可读、可复核”，不是“好看但难验证”。页面应优先服务三类使用者：

- 开发人员：需要快速确认当前 case、输入规模、策略路径、候选结果、错误原因和回退状态。
- 技术评审：需要看到 Agent 是否真的在感知、规划、执行、评估、记忆和回退，而不是静态文案。
- 产品/业务评审：需要用少量业务指标理解系统解决了什么调度问题，但不需要被复杂动效或夸张视觉干扰。

因此，后续所有页面和文档调整都应遵守以下边界：

- 保留 `solver.py` 作为正式评测求解入口，不让网页实验直接改写正式求解链路。
- 保留 Web demo 的实时事件流和 Self-Evolving Code Loop，但把它们改成更清晰的开发者诊断视图。
- 所有本地指标都标注为“本地仿真估算”或“对比参考”，不得包装成官方成绩。
- 视觉风格以中性、克制、高信息密度为主，避免示例图式大渐变、大卡片、夸张背景、强动效和营销化表达。

## 2. 后续需要补齐的内容

### 2.1 场景元数据层

当前 case 列表仍偏技术 ID，例如 `large_seed301`、`tiny_seed42`。后续应增加一层场景元数据，把技术 case 映射为业务和调试都能理解的名称。

建议新增字段：

| 字段 | 用途 | 示例 |
|---|---|---|
| `case_id` | 稳定技术 ID，继续用于 API 和测试 | `large_seed301` |
| `scenario_name` | 面向页面展示的场景名 | `午高峰大规模候选调度` |
| `scenario_type` | 场景类别 | `large_peak`、`low_willingness`、`scarce_couriers` |
| `risk_tags` | 风险标签 | `低接单意愿`、`骑手稀缺`、`合单密集` |
| `operator_note` | 简短场景说明 | `候选行多，适合观察 portfolio routing 和回退策略` |
| `source_type` | 数据来源说明 | `official_case`、`synthetic_demo_case` |

实现位置建议：

- `web_agent_demo/sample_cases.py`：维护 synthetic case 的元数据。
- `web_agent_demo/server.py::list_cases()`：向前端返回展示名、风险标签和来源类型。
- `tests/test_web_agent_demo.py`：断言 case 列表不暴露本地路径，同时包含稳定的场景字段。

验收标准：

- 页面下拉框显示专业场景名，不只显示裸 case ID。
- API 仍保留 `case_id`，避免破坏后端运行逻辑。
- 每个样例至少有一个风险标签和一句场景说明。

### 2.2 风险画像和输入规模摘要

未来页面第一屏不需要像宣传页一样解释项目背景，应直接展示开发者运行前最关心的信息：

- 当前场景名和 case ID。
- 任务数、骑手数、候选行数。
- 平均接单意愿、是否有合单候选、是否骑手稀缺。
- 触发的风险标签，例如低意愿、稀缺、候选爆炸、合单密集。
- 本次运行预算和剩余预算。

这些信息来自现有 `autosolver_agent.system` 中的 case profile 和事件流，不需要引入新依赖。页面表达应简洁，优先使用表格、键值对和状态标签，而不是大面积叙事文案。

验收标准：

- 启动前能看到 scenario metadata。
- 运行后 Case Profile 能展示结构化字段。
- 风险标签与 Agent 事件中的 regime 或 features 一致。

### 2.3 Baseline vs AutoSolver 对比

未来规划中提到的 Before/After 对比应落成一个开发者可验证的结果对比表，而不是夸张的收益展示。

建议对比对象：

- Baseline：`greedy_baseline` 或当前最快稳定基线。
- AutoSolver：本次 Agent 运行最终 best-so-far。

建议对比字段：

| 指标 | 展示目的 | 注意事项 |
|---|---|---|
| 覆盖任务数 | 判断合法覆盖能力 | 不包装成官方分数 |
| 未覆盖任务数 | 暴露风险和失败点 | 可展开任务 ID |
| 使用骑手数 | 判断资源占用 | 与合单/多派相关 |
| 候选组数 | 判断方案复杂度 | 对开发调试有用 |
| 本地相对成本 | 用于候选比较 | 明确标注本地估算 |
| 运行耗时 | 判断预算稳定性 | 使用后端真实 wall time |
| 接受策略数 | 判断 Agent 是否有迭代 | 来自 event stream |

实现位置建议：

- 后端在 report 中保留 baseline summary 和 final best summary。
- 前端新增结果对比区，默认展示关键字段，可展开查看明细。
- 测试中 mock report，断言对比区不会出现“官方成绩”“本地分数”等误导词。

验收标准：

- 用户能一眼看出 AutoSolver 相比 baseline 改善在哪里。
- 表格字段可由当前 report 直接计算或明确标注为估算。
- 没有把 `local_cost` 当作官方成绩展示。

### 2.4 决策解释面板

未来规划中提到的“点击任务组解释决策”可以分两步实现。

第一步先做策略级解释，不要求真实地图交互：

- 当前 Agent 为什么选择这些策略。
- 哪些策略被接受，哪些被拒绝。
- 拒绝原因是质量回退、超时、输出非法还是安全门未通过。
- 当前 best-so-far 从哪个策略产生。

第二步再做任务组级解释：

- 该任务组覆盖了哪些任务。
- 使用了哪些骑手。
- 是否是合单或多派。
- 与替代候选相比的主要差异。
- 是否存在低意愿或骑手冲突风险。

建议先建立策略名称翻译表，让页面减少函数名直出，但保留 hover 或详情里的技术 ID，方便开发人员定位代码。

| 技术策略名 | 页面主标题 | 详情中保留 |
|---|---|---|
| `greedy_baseline` | 快速可行基线 | `strategy_id: greedy_baseline` |
| `single_multidispatch` | 单任务多骑手兜底 | `strategy_id: single_multidispatch` |
| `pair_matching` | 合单机会匹配 | `strategy_id: pair_matching` |
| `sparse_cover` | 稀疏覆盖补全 | `strategy_id: sparse_cover` |
| `production_solver` | 综合求解链 | `strategy_id: production_solver` |
| `evolution_rollback` | 实验策略安全回退 | `event_type: evolution_rollback` |

验收标准：

- 页面主视图使用业务可读标题。
- 详情视图保留技术 ID，便于开发人员查代码。
- 每个 rejected candidate 至少有一条明确原因。

### 2.5 调度结果视图

未来规划中提到“城市网格、订单点、骑手点和派单连线”。这类可视化可以做，但不应做成花哨地图。当前数据本身并不包含真实经纬度，因此第一阶段更适合做抽象调度结果视图：

- 用表格展示任务组、骑手、覆盖任务数、风险标签、来源策略。
- 用简单矩阵或列表展示任务到骑手的关系。
- 若需要示意图，只使用轻量 SVG 或 CSS grid，不引入地图 SDK。
- 明确标注“示意布局，不代表真实地理位置”。

验收标准：

- 没有伪造真实地图或地理含义。
- 开发人员可以从视图直接定位任务 ID 和骑手 ID。
- 可视化不影响事件流、候选表和调试信息的可读性。

### 2.6 ROI 和业务价值估算

ROI 模拟器可以保留为后续增强，但不应放在开发者工作台的主视觉中心。建议放在折叠区域，并明确它是“参数化估算”，不是模型真实线上收益。

输入参数：

- 日订单量。
- 高峰订单占比。
- 单个无人接单订单估算损失。
- 部署商圈数量。
- 成本指数折算系数。

输出字段：

- 预计减少无人接单订单数。
- 预计减少履约损失。
- 预计成本指数下降。
- 参数敏感性说明。

验收标准：

- 所有 ROI 字段都带估算说明。
- 默认不喧宾夺主，不影响开发者调试主流程。
- 参数和公式在页面或文档中可见，避免黑盒化。

## 3. 页面风格准则

后续页面不要参考过于花哨的示例图。建议采用“工程控制台 + 分析工作台”的风格。

### 3.1 视觉原则

- 背景使用浅灰、白色或非常低饱和度底色，不使用大面积彩色渐变。
- 卡片边框、分割线和阴影保持轻量，避免玻璃拟态和强 blur。
- 主色控制在 1 个中性色系和 1 个强调色，例如 slate + blue 或 zinc + green。
- 状态色只用于语义：通过、警告、失败、运行中。
- 动效只用于运行状态提示，禁用大幅位移、呼吸光圈和装饰性动画。
- 字体以系统 UI 字体和等宽字体为主，保证数字、ID、日志可读。

### 3.2 信息架构

建议页面结构调整为：

1. 顶部运行栏：场景选择、预算、运行状态、刷新按钮。
2. 场景摘要：case ID、输入规模、风险标签、数据来源。
3. 结果对比：Baseline vs AutoSolver 的核心指标表。
4. Agent 运行轨迹：阶段轨道、事件流、当前动作。
5. 候选策略表：策略名、技术 ID、耗时、接受状态、拒绝原因。
6. 自进化实验轨道：Generate、Recall、Safety Gate、Sandbox、Decision、Memory。
7. 可选扩展区：调度结果视图、ROI 估算、任务组解释。

这样的结构能让开发人员按“输入 -> 运行 -> 结果 -> 证据 -> 扩展分析”的顺序阅读。

### 3.3 文案风格

- 少用“指挥舱”“黑科技”“降本增效神器”等营销化词汇。
- 多用可验证描述，例如“本次运行接受 3 个候选策略”“实验策略因超时回退”。
- 页面主标题可以是 `AutoSolver Agent Workbench` 或 `调度分析工作台`。
- 中文文案保持短句，日志和 ID 保留英文技术名。
- 所有估算类指标必须带限定词，例如“本地估算”“对比参考”“示意”。

## 4. 分阶段实施路线图

### 阶段一：开发者工作台整理版

目标是在不大改算法的前提下，让页面更专业、更清楚、更适合技术评审使用。

任务：

- 增加场景元数据，替换裸 case 名展示。
- 将首页从展示型 hero 改为紧凑运行栏和场景摘要。
- 保留事件流，但减少装饰性样式和营销化文案。
- 增加 Baseline vs AutoSolver 对比表的前端结构。
- 调整策略候选表，展示技术 ID、可读名称、耗时、状态和拒绝原因。
- 更新 `tests/test_web_agent_demo.py`，覆盖关键文案和误导性词汇禁用。

验收标准：

- 开发人员 30 秒内能确认运行的是哪个 case、什么风险、走了哪些策略、最终结果如何。
- 页面不再依赖大渐变、夸张阴影或炫酷示例图风格。
- 当前功能仍可通过原有 API 运行，不引入第三方前端依赖。

### 阶段二：调度分析增强版

目标是让页面能解释结果，不只解释过程。

任务：

- 在 report 中保留 baseline summary 和 final best summary。
- 增加 rejected candidate 的原因字段和展示。
- 增加任务组级结果表，支持查看任务、骑手、来源策略和风险标签。
- 增加抽象调度关系视图，用列表、矩阵或轻量 SVG 表示，不伪造地图。
- 增加场景之间的指标对比缓存，便于切换 case 后复核。

验收标准：

- 用户能解释为什么采用最终方案。
- 用户能定位某个任务组来自哪个策略。
- 用户能看到失败策略为什么没有进入当前最优解。

### 阶段三：落地接口预留版

目标是为真实业务数据回放和策略治理预留接口。

任务：

- 定义订单、骑手、商圈、天气、时段等外部特征的输入 schema。
- 将本地仿真指标与真实履约指标建立映射表。
- 增加策略版本、策略来源、质量门结果和回退记录的审计字段。
- 增加策略 A/B 对比的离线回放接口。
- 将 Evolution Memory 从展示日志升级为可查询策略库。

验收标准：

- 系统可以接入真实或脱敏业务数据回放。
- 每次策略选择、接受、拒绝、晋升、回退都有审计记录。
- 策略优化结果可以用业务指标和本地评估指标同时解释。

## 5. 开发验证清单

每次推进未来规划相关功能时，至少检查以下项目：

- `python -m unittest tests.test_web_agent_demo`
- `python -m py_compile web_agent_demo/server.py`
- `rg "官方成绩|本地分数|黑科技" web_agent_demo`
- 检查页面是否仍暴露本地文件路径。
- 检查 `local_cost` 是否只作为本地相对比较或内部字段使用。
- 检查新增文案是否同时保留业务解释和技术 ID。

对于样式调整，还需要人工检查：

- 小屏幕下运行栏、场景摘要、事件流和候选表不互相挤压。
- 状态色语义清楚，不能仅靠颜色表达通过/失败。
- 日志、策略 ID、任务 ID、骑手 ID 使用等宽或易读样式。

## 6. 推荐演示流程

未来演示不建议走“炫酷大屏”路线。更专业的流程是按开发者排查路径展开：

1. 选择一个场景，例如 `雨天低接单意愿` 或 `骑手稀缺商圈`。
2. 查看场景摘要，确认任务数、骑手数、候选行数和风险标签。
3. 启动 Agent，观察阶段轨道和实时事件流。
4. 查看候选策略表，确认哪些策略被接受、哪些策略被拒绝以及原因。
5. 查看 Baseline vs AutoSolver 对比，说明最终 best-so-far 的改进来自哪里。
6. 展开 Self-Evolving Code Loop，说明实验策略如何经过安全门、沙箱和回退。
7. 如需讨论业务价值，再展开 ROI 估算，并明确这是参数化估算。

最终希望评审形成的判断是：

```text
AutoSolver Agent 是一个可运行、可观察、可回退的调度优化工作台。
它不是静态求解器包装页，也不是炫技式展示大屏。
```
