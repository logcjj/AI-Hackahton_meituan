# AutoSolver 未来计划

## 总体方向

AutoSolver 后续要从“调度结果展示系统”升级为“AI 可解释调度决策平台”。现有系统已经能在 10 秒内完成配送分配求解并输出可验证派单方案，下一阶段重点不只是继续优化算法分数，而是让评委、业务方和开发人员看到：

- AutoSolver 如何识别场景。
- AutoSolver 如何生成候选方案。
- AutoSolver 如何评估风险。
- AutoSolver 如何淘汰低质量方案。
- AutoSolver 最终为什么选择当前调度路径。

页面风格不参考示例图的花哨大屏方向。实现应保持专业、克制、适合开发人员使用：保留技术 ID、事件证据、估算口径和回退记录，但把算法黑盒拆成可观察、可解释、可复盘的调度决策过程。

本轮参考材料：

- 用户参考图：`/Users/logcjj/Desktop/ae864de6-3c5c-4d85-9aae-2431a6d6737a.png`。参考信息架构，不照搬深色炫酷风格：顶部 KPI、左侧 AI Reasoning Graph、中间 Live Delivery Map、右侧 Decision Explanation、底部 Candidate Strategy Comparison。
- ReasonGraph GitHub：参考其把推理输出转成 sequential/tree reasoning flow 的做法，尤其是节点、边、分支、通过路径和失败/修正路径的表达方式。本项目只借鉴可视化结构，不复制仓库代码或素材。
- AWS `delivery-routes-optimization-for-logistics` GitHub：参考其物流调度域模型和地图层做法，包括 warehouse / vehicle / customer / order / delivery job / route segment、车辆访问链、MapLibre route layer、driver/customer/warehouse markers、VRPTW 约束和距离缓存思路。本项目当前数据没有经纬度，因此先实现抽象路径工作台，不引入 MapLibre 或 AWS 依赖。

最终页面应能同时回答三个问题：

1. AI 是如何思考的？
2. 调度方案如何落地执行？
3. 这个方案能为业务节约多少钱？

## 1. 推理过程可视化升级

后续参考 ReasonGraph 的推理图结构，把一次调度决策拆成完整 AI 推理链路：

1. 输入订单与骑手状态。
2. 场景识别与风险判断。
3. 候选策略生成。
4. 路线可行性校验。
5. 成本与接单风险评估。
6. 最终派单方案输出。

前端展示要求：

- 每条候选方案形成一条可追踪推理路径。
- 正在评估的路径显示为运行中状态。
- 通过验证的路径保持高亮。
- 失败路径灰化并标注淘汰原因。
- 所有节点都保留对应事件、策略 ID 或 report 字段，便于开发人员复核。

第一阶段不做夸张动效，只用状态色、边框和短文案表达运行、通过、失败和回退。这样既能服务答辩展示，也不会牺牲开发调试可读性。

ReasonGraph 参考落地：

- 顺序推理：对应输入、识别、生成、校验、评估、输出六节点。
- 树状/多路径推理：对应贪心基线、合单优先、多派候选、局部修复、最终方案五类候选路径。
- 节点状态：通过路径保留高亮，淘汰路径灰化并写明原因。
- 长推理可读性：节点文本保持短句，技术细节放到事件流和候选表。

## 2. 融合真实物流路径规划界面

后续参考 AWS delivery-routes-optimization-for-logistics 的路线规划界面，把演示页面升级为真实物流调度工作台。地图层或抽象路径层需要展示：

1. 商家位置。
2. 订单配送点。
3. 骑手当前位置。
4. 合单路径。
5. 候选履约路线。
6. 最终选中的派单路线。

当前比赛输入没有真实经纬度，因此第一阶段采用“抽象物流路径工作台”，不伪造真实地图。页面应明确标注该布局是示意视图，用于解释派单关系和履约路径，不代表真实地理位置。

推理图负责解释“为什么这样分配”，路径工作台负责展示“这个方案如何实际履约”。两者联动后，系统形成从算法决策到执行路径的闭环。

当系统选择合单方案时，页面应展示：

- 被选中的骑手和订单。
- 候选路线与最终路线。
- 预计送达时间。
- 合单收益。
- 接单概率估算。
- 被淘汰替代方案的原因。

AWS 路线优化参考落地：

- `warehouse` 对应商家/站点起点。
- `vehicle` 对应骑手或配送资源。
- `customer destination/order` 对应订单配送点。
- `delivery job` 对应最终派单结果。
- `route segment` 对应候选履约路线、合单路径和最终选中路线。
- `distance/time matrix` 的思想用于页面 ETA 和绕行风险估算，但当前只做本地演示估算。

## 3. 候选方案对比与淘汰机制

未来版本必须强化候选策略对比能力。页面应同时展示以下方案类型：

1. 贪心基线。
2. 合单优先策略。
3. 多派候选策略。
4. 局部修复策略。
5. 最终 AutoSolver 方案。

每个方案至少从以下维度对比：

- 覆盖率。
- 无人接单风险。
- 预计成本。
- 骑手占用。
- 预计送达时间。
- 稳定性。

被淘汰的方案要清楚标注失败原因。原因必须翻译成业务和调度语言，而不是只显示“未优于 best-so-far”。常见原因包括：

- 骑手接单意愿不足。
- 路线绕行过长。
- 占用骑手过多。
- 合单收益不明显。
- 无人接单风险偏高。

最终方案以高亮路径保留，并同步显示核心决策依据。这样业务方能理解算法优势来自哪里，而不是只看到一个最终分配结果。

## 4. 商业价值量化

后续系统需要把算法指标转化为业务指标，包括：

- 无人接单减少。
- 骑手利用率提升。
- 履约成本下降。
- 高峰期调度稳定性提升。
- 人工干预减少。

演示口径可以采用以下估算：

- 如果 AutoSolver 相比贪心基线降低约 60% 以上的综合履约成本。
- 在一个日均 10 万单的即时配送城市中，每单节省 0.5 元到 1 元履约损耗。
- 每天可节约约 5 万到 10 万元。
- 按 30 天计算，单城市月度节约空间约为 150 万到 300 万元。
- 扩展到多个城市和高峰时段，月度节约空间可以达到数百万元到千万元级别。

页面必须明确这些数字是“演示口径估算”或“业务价值估算”，不能包装成正式评测结论或真实线上收益。商业价值区应放在决策证据之后，服务说明价值，不喧宾夺主。

- 新增真实物流路径规划工作台的抽象路径视图，覆盖商家、订单、骑手、合单路径、候选路线、最终路线。
