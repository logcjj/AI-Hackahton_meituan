# AutoSolver Agent 产品说明文档

## 1. 产品定位

AutoSolver Agent 是一个面向随机配送分配问题的智能求解系统。它不是只提交一个静态求解器，而是把“候选生成、组合优化、自动评估、预算控制、经验沉淀”组织成可观察的 Agentic Optimization 闭环。

系统在网页端真实启动后端 Agent 控制器，动态回放一次样例求解过程：识别样例画像，进行 regime-aware 策略路由，执行候选方案，由评估器做合法性与相对质量筛选，再根据反馈调整下一轮搜索方向。自进化实验轨道负责展示策略变体生成、安全门、沙箱试跑、回退和候选沉淀。

正式比赛提交仍由稳定的 `solver.py` 完成。网页 Agent 的作用是把求解器背后的 anytime portfolio search、critic-driven selection、自适应策略调度和安全回退过程解释清楚，让评委看到系统具备 AutoSolver Agent 的决策链路，而不是只看到最终输出。

### 阅读说明

为便于技术评审对照页面和代码，本文档采用以下命名约定：

- `solver.py` 指正式评测时执行的求解入口。
- 网页 Agent 指 `web_agent_demo/` 与 `autosolver_agent/` 共同组成的可视化智能体系统。
- 页面中的 `Critic`、`Controller`、`Memory` 分别对应评估器、控制器和记忆模块。
- 页面中的 `Self-Evolving Code Loop` 对应自进化实验轨道。
- 本文档中的“当前最优解”对应页面和代码中的 best-so-far。

## 2. 赛题背景

本题是配送平台中的随机订单分配问题：系统需要把多个配送任务分配给一批可用骑手，并在有限时间内给出合法且质量尽可能好的方案。

输入数据不是简单的“订单 -> 骑手”一对一关系，而是一组带概率和成本属性的候选分配行。每行描述任务、候选骑手、配送相关成本、骑手接单意愿以及是否存在合单可能性。系统需要在概率完成率、期望成本、覆盖约束和骑手资源冲突之间做组合优化。

这个问题的难点在于：高意愿骑手不一定成本低，低成本方案不一定能保证接单，多派可以提高完成概率但也会占用更多骑手资源，合单可能进一步改变组合收益。因此它不是单一贪心可以稳定解决的问题，而是一个带接单概率、合单结构和时间预算约束的多目标分配问题。

AutoSolver Agent 的产品目标，是把这个复杂求解过程组织成可观察的智能体闭环：先抽取样例画像，再进行策略组合调度，自动评估候选方案，根据反馈迭代改进，并在预算结束前保留当前最优合法输出。

## 3. 目标函数与评估边界

本文档说明赛题目标函数相关的工程取舍，不把任何本地计算值当作官方成绩。正式成绩以官方评测和项目中保留的官方提交记录为准。

关键评估维度包括：

- 任务完成概率：一个任务被派给一个或多个骑手后，至少一人接单的概率。多派会提高完成概率，但也会消耗骑手资源，因此需要在 coverage 与 resource contention 之间权衡。
- 期望成本：在不同骑手可能接单的情况下，按接单意愿加权后的成本预期。评估器用本地相对成本比较候选方案是否优于当前最优解。
- 未覆盖惩罚：任务没有被有效覆盖时会带来惩罚。系统会优先保证合法覆盖，再在剩余预算内降低期望成本。
- 时间预算：每个样例需要在有限时间内返回当前最优合法方案。控制器采用 anytime 输出策略，持续维护当前最优解，避免等所有实验结束才输出。

因此，AutoSolver Agent 的核心取舍不是单纯追求“派得越多越好”或“成本越低越好”，而是在完成概率、期望成本、覆盖约束、骑手冲突和时间预算之间做自动平衡。

## 4. 赛题要求对齐

本项目不是只提交一个静态求解函数，而是围绕配送分配问题构建一个可运行、可观察、可验证的 AutoSolver Agent。产品说明按评审最容易检查的能力组织：是否有正式求解器、是否有 Agent 闭环、是否能自主探索策略、是否能自动评估、是否能迭代回退、是否有运行证据。

本项目的对应设计和可见证据如下：

- 不只是一个求解器：保留 `solver.py` 作为正式比赛求解入口，同时提供网页端 Agent 系统展示求解过程。页面阶段轨道、事件流、当前检查器和自进化面板可以直接证明这条链路。
- 自主探索策略：策略规划器会按样例画像选择贪心、多派、匹配、列搜索、流模型、综合求解器等不同策略，形成 regime-aware portfolio。页面策略候选表会展示本轮尝试了哪些策略。
- 自动评估与筛选：评估器判断候选方案是否合法、是否比当前最优解更好，形成 critic-driven selection。页面会展示接受、拒绝和更新当前最优解的状态。
- 迭代改进循环：控制器根据上一轮结果决定继续探索、切换策略、停止或回退，对应 feedback-driven search loop。页面中的阶段变化、预算提示和控制器解释对应这一过程。
- 输出当前最优解：记忆模块始终保留当前最优合法方案，预算结束时直接输出该结果。当前检查器和最终报告会展示这一结果。
- 持续自我进化：Self-Evolving Code Loop 生成策略变体，经过安全门、沙箱、回退/晋升和候选池沉淀，形成安全受控的 strategy evolution channel。不同样例下的 `gen_large_*`、`gen_tiny_*` 截图可以证明面板不是静态装饰。

本文档只引用可以被评委直接复核的材料：页面截图、实时事件流、代码结构、测试体系和官方提交记录。本地运行追踪或页面计算值只用于解释 Agent 决策，不写成官方成绩。

时间约束写法采用保守口径：系统应先快速产生一个合法方案，再在剩余预算内继续改进；正式评测按每个样例 10 秒预算约束；页面和文档展示的是 Agent 如何以 anytime search 方式持续保留当前最优解，而不是等待所有实验完成后才输出。

## 6. 产品结构

AutoSolver Agent 分成两条边界清晰的链路。

下图对应本节结构：左侧是正式比赛求解链路，右侧是网页 Agent 和自进化实验轨道，底部是截图、运行追踪、测试和官方记录构成的证据链。

![AutoSolver Agent 系统架构](../assets/architecture-overview.svg)

| 链路 | 组成 | 作用 |
|---|---|---|
| 正式比赛求解器 | `solver.py` | 官方评测时实际执行的求解入口，保留 `solve(input_text: str) -> list` 接口，目标是在 10 秒预算内返回合法方案。 |
| 网页 Agent 展示系统 | `web_agent_demo/`、`autosolver_agent/`、`tools/`、`tests/` | 用于展示感知、规划、执行、评估、自适应、自进化实验和回退过程。 |

这种设计避免把实验策略直接放进正式评测链路。网页可以展示更完整的 Agent 行为，`solver.py` 仍保持稳定、可提交、可复核。

## 7. 用户能看到什么

网页端重点展示六类信息：

- 样例选择器：切换官方提供样例和演示构造样例，对应 `web_agent_demo/sample_cases.py` 和 `data/official_cases/`。
- Agent 阶段：展示“感知、规划、执行、评估、控制、记忆、自进化”七个阶段，对应 `autosolver_agent/system.py`。
- 实时事件流：按时间顺序显示策略开始、试跑结果、接受/拒绝和回退事件，由后端接口持续推送到页面。
- 当前检查器：展示当前阶段、动作、控制器解释、样例画像和自进化摘要。
- 自进化实验面板：动态回放生成策略、相似历史检索、安全门、沙箱试跑、回退/晋升和记忆状态，对应 `autosolver_agent/evolution.py`。
- 策略候选表：展示候选策略作用、耗时和是否更新当前最优解。

## 8. Agent 核心能力

下图展示 Agent 的核心闭环：先感知样例画像，再规划和执行候选策略，由评估器自动筛选，控制器决定继续迭代或回退，记忆模块保留当前最优方案；自进化实验轨道负责生成和验证可复用策略候选。整体链路对应“感知 -> 规划 -> 执行 -> 评估 -> 控制 -> 记忆”的 Agentic Solver pattern。

![AutoSolver Agent 闭环](../assets/agent-loop.svg)

### 8.1 场景感知

系统会解析输入样例，提取任务数、骑手数、候选行数、平均接单意愿和是否存在合单等特征，并推断样例规模类型。以 `large_seed301` 为例，Agent 识别出的样例画像包含：

```text
规模类型: large
tasks: 40
couriers: 80
rows: 33780
平均接单意愿: 0.299973
存在合单: true
```

这一步对应页面中的 Case Profile 区域，也决定后续策略规划器选择哪些策略。

### 8.2 自主策略探索

Agent 不只调用单一算法，也不是由人手动固定一种方案。它维护一个策略目录，并按样例画像进行 portfolio routing 与试验排序。当前策略包括：

| 页面策略 | 策略类型 | 作用 |
|---|---|---|
| `greedy_baseline` | 快速基线 | 快速得到可行解。 |
| `single_multidispatch` | 启发式策略 | 对单任务尝试多骑手派单。 |
| `disjoint_gain` | 启发式策略 | 按边际收益构造互斥任务组。 |
| `pair_matching` | 匹配策略 | 在存在合单的场景中尝试二元匹配。 |
| `sparse_cover` | 覆盖策略 | 在资源紧张或低意愿场景中优先覆盖高价值任务。 |
| `low_global_column` | 列搜索策略 | 面向低意愿场景构造全局候选列。 |
| `scarce_k2_column` | 列搜索策略 | 面向骑手稀缺场景搜索二任务组合列。 |
| `scarce_bundle_mcf` | 流模型策略 | 用小规模流模型重组稀缺骑手合单。 |
| `production_solver` | 综合求解策略 | 调用 `solver.py` 中完整生产级综合求解链。 |

### 8.3 评估器接受/拒绝

每个候选解会经过评估器检查。评估器关注输出格式、合法性、覆盖情况、本地相对质量和是否优于当前最优解。页面展示接受、拒绝和系统动作，但不把本地计算值当作官方成绩展示。

这一层对应比赛要求中的“自动评估与筛选”：新策略只有在合法且优于当前最优解时才会被保留，否则只作为参考或审计记录保留。

### 8.4 自适应迭代

Agent 会根据上一轮结果决定下一轮动作。

| 观察到的情况 | 后续动作 |
|---|---|
| 低意愿场景 | 转向 low column search，避免继续堆叠普通贪心。 |
| 骑手稀缺场景 | 转向稀缺骑手组合搜索或合单搜索，在合法前提下权衡覆盖与成本。 |
| 覆盖不足 | 转向 sparse cover 和 pair matching 补足任务。 |
| 覆盖完整 | 调用生产级综合求解器做组合搜索与局部改进。 |

### 8.5 当前最优输出

Agent 不需要等到所有候选策略都跑完才输出。控制器和记忆模块会持续维护当前最优解，预算临近时直接输出当前最优合法方案。也就是说，系统采用 anytime optimization 思路：先快速给出可行解，再把剩余时间用于增量改进。

### 8.6 自进化实验轨道

Self-Evolving Code Loop 展示 Agent 的实验轨道。它不直接改写 `solver.py`，而是生成实验策略，经过安全门和沙箱试跑后决定回退或晋升，使策略进化与正式求解链路保持隔离。

| 步骤 | 页面显示 | 系统实际动作 |
|---|---|---|
| Generate | 本次生成的策略变体 ID，例如 `gen_large_v044` | 在实验目录中生成一个新的 Python 策略文件。 |
| Recall | 是否命中相似历史候选 | 从历史候选池中查找同类型、相似样例画像的可信候选。 |
| Safety Gate | 安全门是否通过 | 检查代码结构、导入模块、危险调用和固定接口。 |
| Sandbox | 失败原因或试跑结果 | 在短时间窗口内运行实验策略，识别超时、格式错误或质量回退。 |
| Decision | 回退或晋升动作 | 失败则回退到稳定基线方案，成功则晋升候选池。 |
| Memory | 审计和候选池状态 | 所有结果写入审计，只有可复用策略进入候选池。 |

## 10. 使用方式

启动本地网页：

```bash
cd /Users/比赛/FOR_AutoSolver_706.20_提交版
python3 web_agent_demo/server.py --host 127.0.0.1 --port 8765
```

打开浏览器：

```text
http://127.0.0.1:8765
```

推荐演示流程：

1. 选择 `large_seed301`，点击 `启动 Agent 求解`。
2. 观察事件流如何从“感知”进入“规划、执行、评估”。
3. 向下滚动到 Self-Evolving Code Loop，查看生成策略、安全门、沙箱结果、回退动作和记忆状态。
4. 切换 `tiny_seed42` 或 `low_willingness_seed501` 再运行一次，观察样例画像和策略 ID 如何变化。
5. 强调正式提交仍由 `solver.py` 完成，网页 Agent 用于解释系统能力和迭代过程。
6. 说明系统不只是结果展示页，还包含 Agent 自主探索、自动评估、迭代改进和技术证据链。

## 11. 与普通展示网页的区别

| 普通展示网页 | AutoSolver Agent |
|---|---|
| 主要展示固定文案和静态结论 | 每次运行都会产生实时事件流和动态自进化回放。 |
| 用户只能看结果 | 用户能看到感知、规划、执行、评估、控制、记忆和自进化过程。 |
| 样例切换不影响核心解释 | 不同样例触发不同样例画像、策略路线和实验策略 ID。 |
| 失败原因容易混在一起 | Sandbox 显示真实失败原因，Decision 单独显示系统回退动作。 |
| 容易把页面指标误当官方成绩 | 页面与文档明确区分运行解释、评估器信号和官方提交记录。 |

![AutoSolver 文档证据链](../assets/verification-evidence.svg)

正式比赛结果只引用项目中保留的官方提交记录。该记录显示：

```text

```

产品边界说明：

- 网页 Agent 展示系统不在正式评测时运行。
- 自进化记忆区不进入 `solver.py` 的 10 秒正式求解链路。
- 生成策略失败时只写入审计日志，不进入候选池。
- `solver.py` 是稳定提交物，网页实验不会自动改写它。
- 页面本地计算值只用于解释 Agent 决策，不作为官方成绩。
- 如评审涉及成绩口径，以官方最终评测规则和官方提交记录为准。

综上，AutoSolver Agent 的展示重点不是把页面当作成绩面板，而是让评委看到系统如何围绕配送分配问题进行感知、规划、自动评估、迭代改进、安全回退和经验沉淀；正式求解结果则由稳定的 `solver.py` 和官方提交记录支撑。
