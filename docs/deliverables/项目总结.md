# AutoSolver Agent 项目总结

## 1. 项目背景

AutoSolver Agent 面向美团即时配送中的智能派单与履约优化问题。即时配送的核心挑战在于：平台需要在极短时间内，把大量订单合理分配给骑手，并尽可能保证订单被接单、被履约，同时控制平台成本、骑手资源占用和用户等待风险。

在真实业务中，派单并不是简单的“把订单分给最近的骑手”。一个订单能否被顺利履约，受到多种因素共同影响：骑手是否愿意接单、该骑手是否已经被其他任务占用、是否存在合单机会、当前区域是否骑手稀缺、天气或高峰期是否导致接单意愿下降、平台是否需要在完成率和履约成本之间做权衡。

因此，这类问题本质上是一个带概率、成本、资源约束和时间约束的组合优化问题。系统需要在大量候选派单方案中选择一个合法组合，使整体期望履约效果尽可能好。

本项目的输入可以理解为一张候选派单表。每一行代表一个已经生成好的可选执行方案：

```text
任务或任务组 + 候选骑手 + 履约成本 + 骑手接单意愿
```

其中，任务可以是单个订单，也可以是合单任务组；骑手接单意愿可以理解为该骑手接受该任务的概率；履约成本用于衡量该派单方案的代价。系统需要从这些候选行中选择一批输出方案。

输出结果则是：

```text
某个任务或任务组 -> 分配给哪些骑手
```

这意味着系统需要同时处理三类决策：

- 是否给一个任务多派多个骑手，以提高至少一人接单的概率。
- 是否选择合单任务组，让一个骑手覆盖多个任务。
- 如何避免骑手重复使用、任务冲突和无效派单。

在这个背景下，AutoSolver Agent 的目标不是单纯做一个算法函数，而是构建一个面向即时履约场景的智能调度 Agent。它需要具备感知当前场景、选择合适策略、生成候选方案、自动评估质量、保留当前最优解、安全回退和解释决策的能力。

## 2. 项目定位

本项目采用“正式求解器 + Agent 展示系统”的双层架构。

正式求解器负责在比赛评测约束内输出高质量合法方案。它保持单文件、无第三方依赖、限时返回，适合进入官方评测热路径。

Agent 展示系统负责把求解过程产品化、可视化和可解释化。它不是替代正式求解器，而是将求解器背后的策略选择、自动评估、当前最优维护和安全回退过程展示出来，让评委能够看到系统具备完整的 Agent 行为，而不是只看到最终输出结果。

项目可以概括为：

```text
一个面向即时配送履约场景的调度优化 Agent，
能够在复杂候选派单空间中自动搜索、评估并输出当前最优派单方案。
```

从业务视角看，它可以被理解为“即时履约调度指挥舱”的核心算法与 Agent 原型；从技术视角看，它是一个限时组合优化求解器加一个可观察的 Agent 控制系统。

## 3. 核心算法

### 3.1 问题建模

项目将派单问题建模为候选方案选择问题。每个候选方案包含：

- 任务集合：可能是单个任务，也可能是多个任务组成的合单。
- 骑手 ID：表示该候选方案由哪位骑手执行。
- 履约成本：代表该方案的成本或代价。
- 接单意愿：代表该骑手接受该任务或任务组的概率。

求解器需要选择若干候选方案，使整体方案满足以下约束：

- 同一个骑手不能在最终方案中重复使用。
- 任务覆盖需要尽可能完整。
- 多派可以提升任务完成概率，但会占用更多骑手资源。
- 合单可能降低骑手占用，但也会改变成本和风险结构。
- 系统必须在有限时间内返回可用结果。

评估时，系统关注的是期望履约成本。简单来说，如果至少有骑手接单，则根据接单骑手的成本计算期望代价；如果无人接单或任务未覆盖，则产生较高惩罚。因此，接单概率不是一个孤立加分项，而是通过降低无人接单风险影响最终方案质量。

### 3.2 限时 Portfolio Solver

正式求解器不是单一贪心算法，而是一个限时 portfolio solver。所谓 portfolio solver，是指系统不会依赖一种固定策略，而是根据当前样例特征，在时间允许的情况下串行尝试多种启发式策略，并从它们生成的候选方案中选择当前最优。

求解器首先识别当前场景特征，例如：

- 任务数量。
- 骑手数量。
- 候选行数量。
- 平均接单意愿。
- 是否存在合单候选。
- 是否属于骑手稀缺场景。
- 是否属于低接单意愿场景。
- 是否属于小规模、中规模或大规模场景。

然后，系统根据这些特征选择不同策略组合。

普通场景下，系统会优先快速生成可行解，再尝试多派、合单匹配和局部优化。

低接单意愿场景下，普通贪心容易不断增加派单但收益有限，因此系统会转向低意愿专用搜索和更稳健的风险控制策略。

骑手稀缺场景下，系统不能盲目追求全覆盖或多派，因为每个骑手都是稀缺资源。此时系统会更加重视稀疏覆盖、二任务组合、合单重组和骑手资源再分配。

大规模场景下，系统会先用轻量策略快速建立 baseline，再调用综合求解链和局部修复策略，在剩余时间内继续降低期望成本。

### 3.3 主要策略

项目中使用的策略可以分为以下几类。

第一类是贪心基线。它用于快速构造一份可行解，让系统在很短时间内获得一个可以比较和回退的 baseline。这个策略不是最终能力上限，而是保证 anytime 行为的基础。

第二类是单任务多派。它只考虑单任务候选，通过给关键任务选择多个不同骑手，提高至少一人接单的概率。该策略适合骑手相对充足、需要提高履约稳定性的场景。

第三类是合单匹配。它处理多个任务由同一骑手执行的候选方案，判断合单是否比拆成多个单任务更划算。该策略能够减少骑手占用，并在合单结构明显的场景中提升整体方案质量。

第四类是互斥任务组启发式。它优先选择边际收益高且彼此不冲突的任务组，避免多个候选之间重复覆盖或互相干扰。

第五类是低接单意愿搜索。它面向雨天、恶劣天气、低意愿时段等场景，重点控制无人接单风险，避免普通贪心在低概率候选上消耗过多资源。

第六类是骑手稀缺搜索。它面向骑手数量不足或供需紧张的区域，通过稀疏覆盖、二任务组合列搜索、小规模流模型和 bundle 重组，在有限骑手资源下获得更好的整体履约效果。

第七类是局部修复和交换。系统在选出当前最优方案后，还会在剩余时间内继续尝试换骑手、拆合单、合并任务组、删去低收益组、pair/triple exchange、窗口修复和重新分配，以进一步降低期望成本。

这些策略共同构成一个“先快速可行，再逐步变好”的求解过程。

## 4. Agent 机制

项目不仅包含求解算法，还将求解过程组织成 Agent 闭环。Agent 的职责不是简单调用一个函数，而是围绕一个具体派单场景完成完整决策流程。

Agent 首先进行场景感知。它读取候选数据，识别任务数、骑手数、候选规模、接单意愿分布、合单机会和当前场景类型。

随后进入策略规划。系统根据场景类型选择初始策略组合。例如，大规模或中等规模场景会尝试合单匹配；低意愿场景会转向低意愿搜索；骑手稀缺场景会转向稀缺骑手专用策略。

执行阶段会串行试跑多个候选策略。每个策略都有自己的时间片，并且每次运行前都会检查剩余时间。如果时间不足，系统会停止尝试新策略，直接保留当前最优解。

评估阶段由 Critic 完成。Critic 会判断候选方案是否合法、覆盖情况如何、期望成本是否优于当前 best-so-far。只有合法且更优的候选才会更新当前最优方案。

控制阶段会根据当前最优方案的状态决定下一步。如果覆盖不足，Agent 会尝试补覆盖策略；如果覆盖完整，Agent 会调用生产级综合求解器；如果是低意愿或骑手稀缺场景，则进入对应专用策略。

记忆阶段会持续保存当前最优解，保证系统具备 anytime 输出能力。即使后续实验策略失败，系统也能回退到已经验证过的稳定方案。

自进化实验轨道用于展示策略生成、安全门、沙箱试跑和回退机制。实验策略不会直接修改正式求解器，只有通过安全检查和质量门的策略才可能进入候选记忆。这保证系统具备探索能力，同时不会污染生产求解链路。

## 5. 解决的具体问题

AutoSolver Agent 解决的不是单一算法题，而是即时履约场景中的一组实际运营问题。

第一，解决复杂候选空间下的快速决策问题。真实派单场景中，一个任务可能对应多个骑手，一个骑手也可能对应多个任务组，合单、多派、成本和接单概率交织在一起。系统需要在大量候选行中快速选择合法组合。

第二，解决低接单意愿带来的履约风险问题。在雨天、高峰、距离较远或订单吸引力不足的场景中，低成本方案未必能被接单。系统通过多派、低意愿搜索和期望成本评估，降低无人接单风险。

第三，解决骑手稀缺下的资源分配问题。当可用骑手少于任务需求时，盲目追求全覆盖可能导致成本过高或方案不稳。系统会使用稀疏覆盖、合单和局部修复，在有限骑手资源下做整体最优权衡。

第四，解决合单与单派之间的选择问题。合单能够减少骑手占用，但并不总是更优。系统会通过候选评估和匹配策略判断何时应该合单，何时应该拆成单任务。

第五，解决有限时间内持续优化的问题。正式评测和真实业务都要求系统不能长时间搜索。AutoSolver Agent 采用 anytime 思路，先快速生成可行解，再在剩余时间内持续优化，并随时保留当前最优方案。

第六，解决复杂算法不可解释的问题。网页端 Agent 把求解过程拆解成感知、规划、执行、评估、控制、记忆和回退，使评委和业务方能够理解系统为什么选择某个方案、为什么拒绝某个策略、为什么最终采用当前方案。

## 6. 项目价值

从平台角度看，AutoSolver Agent 有助于降低无人接单和履约失败风险，减少不必要的派单成本，并提升高峰期调度稳定性。

从用户角度看，更稳定的派单方案意味着更低的超时风险和更确定的服务体验。

从骑手角度看，系统可以减少低质量或不合理派单，提高任务与骑手意愿之间的匹配度。

从商家角度看，履约稳定性提升有助于降低订单取消、延迟和用户投诉。

从运营角度看，Agent 不仅给出结果，还能展示场景风险、策略选择、方案对比和回退过程，为人工运营提供可理解的辅助决策工具。

因此，本项目的核心价值不是“实现了一个更复杂的算法”，而是将即时履约派单问题从单点求解升级为一个可运行、可解释、可回退、可演示的智能调度 Agent 原型。
