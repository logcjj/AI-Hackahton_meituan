# Goal 2 Plan

## 原始反馈

用户指出上一轮实现没有真正参考“我的未来计划”，随后直接给出完整《AutoSolver 未来计划》。本轮必须以这份用户直接提供的未来计划为权威来源。

## 核心目标

1. 以 `goal/goal-2/input.md` 中用户直接提供的《AutoSolver 未来计划》为权威来源。
2. 逐条提取未来计划中的明确要求：ReasonGraph 推理过程、真实物流路径规划界面、候选方案对比与淘汰机制、商业价值量化。
3. 对照当前 Web demo 和文档，识别漏做、偏离或误读的部分。
4. 修正文档和页面，让结果能被证据证明“确实参考了未来计划”。
5. 保持用户要求的专业、开发者适用、不过度花哨风格。

## 风险

- 未来计划现在已有用户直接提供版本，历史版本只能作为背景，不能覆盖用户最新输入。
- 我之前已经提交过一轮改动，必须避免用已有结果反向定义目标。
- 当前分支已有 4 个本地提交，修正应增量完成，不随意重写历史。
- 本地 Web demo 运行会写 Evolution 状态文件，验证后必须清理副作用。

## 执行方案

1. 将用户最新未来计划完整保存到 `goal/goal-2/input.md`。
2. 建立逐项映射表：未来计划原文条目 -> 当前实现证据 -> 缺口 -> 修正动作。
3. 修正 `docs/deliverables/未来规划.md`，保留用户计划的四大方向，并增加开发实现口径。
4. 修改 Web demo：增加专业克制的 ReasonGraph 决策链、物流路径工作台、候选方案淘汰对比、商业价值量化区域。
5. 运行测试、语法检查、静态检查和必要页面检查。
6. 更新 `tasks.md`，提交本轮修正。

## 验证方式

- `goal/goal-2/input.md` 完整保存用户最新未来计划。
- 对每个未来计划条目给出当前实现证据或明确修正。
- 页面中能检索到 `ReasonGraph`、`真实物流路径规划`、`候选方案对比与淘汰`、`商业价值量化`、`AI 是如何思考的`、`调度方案如何落地执行`、`这个方案能为业务节约多少钱`。
- 运行 `python3 -m unittest tests.test_web_agent_demo`。
- 运行 `python3 -m py_compile web_agent_demo/server.py web_agent_demo/sample_cases.py autosolver_agent/system.py tests/test_web_agent_demo.py`。
- 检查工作区无测试副作用。

## 回滚方案

- 用 `git diff` 只回滚本轮新增修正，不使用 destructive git 命令。
- 不回滚用户已有提交或无关改动。
