# Plan

## 需求

本轮目标是把 AutoSolver 演示页从“算法结果展示 + 局部模拟地图”重做成更真实的 To B 调度演示工作台。核心要求：

1. 设计 5-6 类业务场景，每类场景内可刷新生成 10 条模拟样本。
2. 刷新后先展示商家/订单点与骑手点，不展示最终派单线，符合“尚未推理”的状态。
3. 点击开始推理后，左侧 ReasonGraph/策略链路按场景动态选择 5 条策略之一或多条策略组合，不同场景不能总走同一条路径。
4. 推理完成后，地图刷新并显示每个商家/订单派给哪个骑手，可为每个商家连 1-2 条派单线。
5. 地图需要像真实导航/城市商圈图层，接近百度地图/导航图层的层级感：道路、街区、建筑块、路口商圈点位、骑手沿道路分布、路况热度/拥堵感；不需要真实路名和精确地理位置，也不要标具体是哪条路。
6. 每个场景至少 3-4 个商家/订单点、多个骑手；最多可到 5-6 个商家/订单和十几个骑手，不能只有一个商家和两个骑手。
7. 意愿、价格、距离、ETA、风险等模拟数据要与位置和场景有关，并能驱动派单算法/策略展示。
8. 点击骑手、订单、线路后，右侧显示对应位置、订单期望、派单依据、接单概率、成本/ETA/风险。
9. 所有按钮和交互必须可用，不能只是 toast 或视觉摆设。
10. 通过严格浏览器测试、截图和 DOM 审计，达到高级真实项目演示标准。

## 上下文

当前主要代码在 `web_agent_demo/server.py`，测试在 `tests/test_web_agent_demo.py`。前一轮已做过：

- 初始状态无路线和派单详情。
- 根据 case 候选数据生成模拟坐标。
- 推理后在 SVG 地图上展示 pin、label、dispatch-link。
- 默认聚焦当前 assignment，候选线低噪显示。
- 仍存在不足：地图仍不像真实导航/商圈图层，缺少百度地图式道路/建筑/路况层级；场景样本数量不足，刷新不是“生成新样本”，策略链路仍不够随样本变化，派单线和右侧详情还不满足用户的新标准。

## 风险

- 只做视觉会再次被认为“形而上”；必须让刷新、推理、策略、地图、详情都由同一份场景样本数据驱动。
- 如果完全依赖真实算法输入文件，缺少商家/骑手坐标，地图仍会显假；需要在前端/后端生成可复现的仿真商圈数据。
- 如果样本随机不可控，测试不稳定；需要 deterministic seed，并支持刷新切换 10 个样本。
- 如果每个样本都跑真实求解器，耗时和数据结构可能不适合演示；应把真实算法 report 与仿真场景派单层解耦，同时保留算法口径。
- 如果路线过多会拥挤；需要默认可读、可点击、可切换图层。

## 执行方案

1. 建立新的仿真数据模型：scenario type、sample index、merchant/order points、courier points、candidate costs/willingness、recommended strategy、assignment result。
2. 为每类场景生成 10 个 deterministic samples，刷新按钮在当前场景内切换 sample，而不是只重新加载 case list。
3. 重构地图状态：`preview` 状态只显示商家/订单和骑手；`reasoning` 状态显示策略评估进度；`final` 状态显示派单线和 assignment detail。
4. 重做仿真地图底图：街区、建筑块、主次道路、路口商圈热区、骑手沿道路分布、路况热度/拥堵感；风格参考百度地图/导航图层的层级，但所有路名和真实地址都隐藏。
5. 将 5 条策略映射到场景和样本特征：合单优先、多派候选、局部修复、贪心基线、风险平衡，每条策略都至少能在某些样本中成为 selected/evaluating/rejected。
6. 派单线按道路感折线连接，所有商家/订单都有最终派单线；当前选中线高亮，候选/其它派单线低噪但可见。
7. 完善右侧详情：点击骑手、订单、线路、商家后显示对应实体与 assignment 的价格、ETA、接单概率、风险、策略理由。
8. 完善按钮：刷新样本、运行推理、图层切换、只看最终、候选/全部、定位当前派单、适配视图、放大缩小、全屏聚焦、场景切换都要有真实状态变化。
9. 每三个 task 后做大型检查-debug 循环：语法、单测、JS、浏览器截图、全场景样本抽样、交互点击审计。

## 验证方式

- `python3 -m py_compile web_agent_demo/server.py tests/test_web_agent_demo.py web_agent_demo/delivery_routes_clone.py web_agent_demo/reasongraph_clone.py`
- `python3 -m unittest tests.test_web_agent_demo`
- `node --check` 检查内联前端脚本。
- 浏览器打开 `http://127.0.0.1:8765/`，对 5-6 个场景分别点击刷新、运行、地图实体、线路和图层按钮。
- DOM 审计每个场景至少有 3 个商家/订单点、多个骑手点、运行后每个商家/订单有 assignment 线。
- 截图检查：初始/刷新状态无最终线；运行完成状态有清晰派单线；地图有导航图层式道路、建筑、路口商圈和路况层级，但不显示真实路名；整体像 To B 调度工作台而不是随机线条。
- 策略审计：不同场景 selected branch 不完全相同，5 条策略都能在样本集合中被触达。

## 回滚方案

所有改动通过 `git diff` 定位；若新仿真层引入问题，用 `apply_patch` 回退 `web_agent_demo/server.py` 的新增数据模型、地图渲染和测试断言。不使用 `git reset --hard` 或 `git checkout --`。
