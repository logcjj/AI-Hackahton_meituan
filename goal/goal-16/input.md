# Original Input

/goal 创建一个新的分支，你现在要为一个“外卖配送智能调度演示系统”重新设计并实现整套前端。

这次不是普通参考，也不是在现有页面上修补，而是要你**直接逆向一个成熟调度产品的前端结构、页面拆分、交互逻辑和视觉风格**，然后把其中的业务对象替换成我们的业务。

一、唯一参考方向
请直接逆向参考 Kandbox Dispatch 这一整套前端产品，而不是只参考它某一个页面。https://kd1.kandbox.tech/

你要逆向和学习的是：
1. 它的页面拆分方式
2. 它的工作台结构
3. 它的布局层级
4. 它的交互逻辑
5. 它的视觉系统
6. 它是如何把资源、任务、规划、地图、评分这些模块拆开的

参考对象：
- Kandbox Dispatch Live Site
- Kandbox Dispatch Docs / Guide
- 公开可见的前端结构、页面路由、页面模块
- 如果可行，可以直接从该项目公开仓库、公开页面、公开资源中反推它的前端组织方式

你学习的不是“某个地图样式”，而是它整套产品壳子和前端骨架。

二、最重要的要求
1. 完全抛弃当前项目已有前端，不要延续任何旧页面结构
2. 不要基于现有页面修修补补
3. 直接按照 Kandbox Dispatch 那种“多页面调度工作台”的拆法重做
4. 我们的业务内容替换进去，但产品拆法、模块分层、工作台思维要向它靠拢
5. 设计风格必须简洁、专业、高级、克制，像成熟企业级调度系统
6. 不重要的东西不要展示，只展示能体现我们优势的内容

三、你要逆向的核心产品模式
Kandbox Dispatch 这类系统本质上不是一个“单页大地图”，而是一整套调度工作台。
它通常会拆成类似下面的模块：
- Workers
- Jobs / Orders
- Planner / Chart / Gantt
- Live Map
- 评分卡
- 弹窗/抽屉/详情模块

请你学习这种拆法，然后将其替换为我们的版本：

Kandbox 的模块 -> 我们的模块
- Workers -> 骑手页
- Jobs / Orders -> 订单页
- Planner / Chart -> 决策页
- Live Map -> 实时推理页
- Planner Score -> 算法对比评分卡
- 历史/调度辅助信息 -> Memory 页

四、我们的产品目标
做一套“外卖配送实时智能调度工作台”，重点突出：
1. 实时推理过程
2. 订单、骑手、地图的联动
3. 我们的算法相对基线算法的累计优势
4. 决策过程可追溯
5. Memory（沉淀记忆）可视化

五、必须拆成的主页面
请将系统拆成以下 5 个主页面：

1. 实时推理页
2. 订单页
3. 骑手页
4. 决策页
5. Memory 页

六、每个页面的明确要求

A. 实时推理页
这是系统主页面，请参考 Kandbox 的 Live Map / GanttMap 类工作台方式来设计，但业务内容换成我们的。

目标：
展示一整天配送过程中，系统如何自动进行实时推理、实时分配、实时累计对比。

关键要求：
1. 页面有一个“开始推理”按钮
2. 点击一次后，系统自动开始推进整天推演
3. 后续不需要用户一轮一轮手动点击
4. 支持暂停、继续、倍速
5. 推理过程自动展示：订单进入、候选分配、路线变化、骑手移动、累积结果变化

这一页必须既有地图，又有对比评分卡，而且在同一个页面里。
页面可以上下滚动，但主体验要像一个成熟工作台，不像 PPT，也不像单张大图。

页面结构建议：
1. 顶部固定控制条
   - 开始推理
   - 暂停/继续
   - 倍速
   - 模式切换（当前算法 / 对比 / 叠加）
2. 中间主地图
   - 商家
   - 订单
   - 骑手
   - 路线
   - 热点区域
3. 右侧或浮动位置：实时累计对比栏
4. 下方：轻量事件流 / 本轮摘要 / 累积指标区

实时累计对比栏必须突出展示：
- 基线/弹金算法累计
- 我们的算法累计
- 时间差异
- 金钱差异
- 超时单差异
- 空驶里程差异
- 收益/成本差异

展示方式必须：
- 动态实时变化
- 简洁
- 清楚
- 美观
- 不要大表格
- 不要信息过载

地图动作要求：
- 地图主要展示我们的算法的实时动作
- 订单进入时有克制的动态效果
- 骑手沿路径平滑移动
- 路线重分配时旧路线淡出、新路线接管
- 对比模式下不要把两套路线全铺满地图
- 如果做叠加，只强调差异部分

这一页最重要的视觉目标是：
“用户一眼就能看出系统正在自动实时推理，并且我方累计结果持续优于基线算法。”

B. 订单页
请参考 Kandbox 的 Jobs / Orders 类页面，但替换为我们的外卖订单体系。

要求：
1. 一整天的订单数据提前预置进去
2. 不要做成需要用户手工维护的后台录入页
3. 页面展示全天订单全集，并支持时间段、商圈、状态筛选
4. 页面气质是调度输入视图，而不是 CRUD 表管理

建议展示内容：
- 订单编号
- 商家/提货点
- 下单时间
- 承诺送达时间
- 当前状态
- 风险等级
- 所属商圈
- 是否已进入推理
- 基线算法结果
- 我方算法结果

C. 骑手页
请参考 Kandbox 的 Workers 类页面，但替换为我们的骑手资源体系。

要求：
1. 一整天的骑手数据提前预置进去
2. 展示班次、状态、位置、负载、任务链
3. 页面是资源盘点视图，不是人事后台

建议展示内容：
- 骑手编号/名称
- 在线状态
- 班次时间
- 所属区域
- 当前负载
- 当前任务链
- 预计空闲时间
- 历史表现摘要
- 小地图位置预览

D. 决策页
请参考 Kandbox 的 Planner / Chart / Gantt 这类“规划视图”的产品角色，但不要简单做排程表，要做成我们的“推导过程页”。

这个页面的目标：
完整记录和展示每一轮推理/决策是如何产生的。

不是点某个骑手后才看局部详情，而是独立成一个完整页面。

每条决策记录都建议包含：
- 触发时间
- 触发原因
- 输入订单集合
- 候选骑手集合
- 过滤过程
- 评分过程
- 最终动作
- 被放弃动作
- 本轮结果
- 结果回写

页面结构建议：
1. 左侧：决策轮次时间线
2. 中间：当前轮推导过程
3. 右侧：输入上下文 + 输出结果 + 本轮摘要

E. Memory 页
这里不要做成“资产化”。
这里必须参考 Hermes 风格去理解：
我们沉淀的是“记忆”，不是资产表，不是资料库。

这个页面的目标：
展示系统在历史推理中沉淀下来的 Memory，以及当前如何召回和使用这些 Memory。

建议展示结构：
1. 新沉淀记忆
2. 已整理记忆
3. 当前命中的记忆
4. 记忆效果反馈

每条 Memory 建议字段：
- 触发场景
- 上下文摘要
- 策略摘要
- 决策结果
- 效果反馈
- 置信度
- 召回次数
- 最近命中时间

页面气质：
要像“系统长期记忆视图”，不是日志列表，也不是文档中心。

七、数据约束
1. 订单页和骑手页的数据必须提前预置好
2. 预置的是一整天全量数据
3. 实时推理页只是按时间推进，逐步释放和消费这些数据
4. 决策页和 Memory 页也基于这些推理过程自动生成内容
5. 不要把“数据维护”作为主叙事

八、视觉系统要求
请整体参考 Kandbox Dispatch 这种成熟调度产品的气质，但不要机械照抄。
必须做到：
1. 简洁专业
2. 企业级工作台风格
3. 高信息密度但不杂乱
4. 悬浮卡片、抽屉、时间线、地图层、轻量图表协同
5. 不要夸张动画
6. 不要游戏 UI
7. 不要大屏展示风
8. 不要幼稚化
9. 不要无意义装饰

关键词：
专业、克制、现代、可信、简洁、高级、调度工作台

九、必须避免的错误
1. 不要继续沿用当前项目前端
2. 不要只做单页大地图
3. 不要做成一堆普通后台表格
4. 不要堆满信息
5. 不要让用户频繁点击推进
6. 不要把 Memory 做成“资产化”
7. 不要只做视觉模仿，必须模仿的是“产品结构和工作台拆法”
