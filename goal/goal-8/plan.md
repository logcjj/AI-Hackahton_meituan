# Plan

## 需求

本轮目标不是继续在现有页面上做小修，而是把 AutoSolver 演示工作台往更专业的 To B 美团即时配送调度产品靠拢：

1. 参考外部 GitHub / deck.gl / To B 地图项目，提炼专业地图、线路、图标、纹理和控制按钮风格。
2. 地图不应每次“乱随机”，而应先设计合理道路与位置锚点：商家可出现点、骑手可出现点、商圈/雨天/稀缺/活动等场景的点位范围。
3. 增加或完善“刷新地图/刷新位置”能力，让同一场景基于预设锚点随机抽样，而不是固定或无约束随机。
4. 提前准备 10 个调度场景，每个场景有合理的商家与骑手位置分布、天气/密度/风险特征。
5. 骑手到商家的派单线必须更像沿道路连接，不能像不贴路的折线或断开线。
6. 整体视觉更偏 To B、专业、克制，适合企业/开发人员演示，避免花哨炫酷。
7. 图标、按钮、箭头、线路和地图纹理资源需要统一升级，尽量参考美团点餐/配送地图语境。
8. 所有新增按钮和交互必须可用，并通过浏览器自测。
9. “刷新骑手和商家位置”与“刷新地图”必须是两个不同能力：前者重抽当前场景实体锚点和候选关系，后者切换底图/道路/纹理/交通可视变体，不应混成一个按钮或同一语义。

## 上下文

当前前端主要在 `web_agent_demo/server.py` 内生成单页 HTML/CSS/JS。上一轮 `goal-7` 已完成工作台固定布局、场景入口去重、基础按钮可用和 10 秒推理流程，但用户仍指出：

- 地图底图、纹理、线路和 To B 质感仍不足。
- 商家和骑手位置没有“预先描定”的业务合理性。
- 路线没有真正顺道路连上。
- 需要参考 deck.gl TripsLayer 或其他 GitHub 项目的成熟地图表达，而不是守着现版本。

本轮应把地图/点位/线路抽象升级为“预设路网 + 锚点池 + 场景采样 + 沿路派单线 + To B 视觉资源”的体系。

## 调研方向

- deck.gl TripsLayer：学习时间序列轨迹、路线宽度、颜色、淡入淡出和路线数据结构。
- vis.gl / deck.gl 示例：参考地图叠层、控制面板和数据驱动渲染方式。
- 开源物流/配送地图项目：参考路线规划、车辆/配送点图标、暗色底图和控制栏。
- Mapbox / Kepler.gl / Uber 可视化风格：提炼 To B 数据地图的克制配色和层级。
- 美团配送/外卖语境：使用黄色商家、蓝绿骑手、橙色风险/天气提示，但避免品牌化过度。

## 执行方案

1. 外部调研并形成可落地设计清单，明确哪些要引入当前项目，哪些不适合。
2. 审计当前代码的数据流：场景生成、地图层、实体点、路线生成、按钮和浏览器测试。
3. 新增位置锚点体系：地图道路节点、商家锚点、骑手锚点、场景配置与采样规则。
4. 扩展为 10 个场景：商圈高峰、雨天低意愿、骑手稀缺、活动压力、低峰分散、夜间商圈、学校/园区、医院/写字楼、交通拥堵、异常补单等。
5. 增加“刷新地图/刷新位置”按钮或明确现有刷新按钮能力，让刷新只在当前场景锚点池内抽样。
   - 拆分为两个按钮：`刷新位置` 用于当前场景内重抽商家/骑手锚点；`刷新地图` 用于更新底图纹理、道路交通/地图变体和视觉状态。
6. 重构派单线路：基于路网节点和锚点最近路段生成 Manhattan/道路段路径，箭头放在道路中段，长距离低噪。
7. 统一视觉资源：To B 色板、地图纹理、按钮状态、商家/骑手图标、线路和箭头样式。
8. 浏览器执行严格验收：10 个场景切换/刷新/运行、地图按钮、点线详情、截图与审计 JSON。

## 风险

- 直接引入 deck.gl 可能带来构建、依赖和 WebGL 兼容成本；若当前项目没有打包链路，应优先借鉴数据结构和视觉表达，而不是硬接复杂依赖。
- 接入外部瓦片/纹理可能受网络、版权、稳定性影响；需要优先使用可控本地或 CSS/SVG 程序化纹理。
- 10 个场景和锚点体系如果一次性改完，容易引入大量逻辑回归；应分任务推进，每个任务都浏览器验证。
- 如果只做视觉不做锚点/线路，会继续被认为“形而上”；如果只做算法不改视觉，也不满足 To B 观感。

## 验证方式

- `python3 -m py_compile web_agent_demo/server.py tests/test_web_agent_demo.py`
- 提取内联脚本后 `node --check /tmp/autosolver-inline.js`
- `python3 -m unittest tests.test_web_agent_demo`
- `python3 -m unittest`
- 浏览器打开 `http://127.0.0.1:8765/`
- 对 10 个场景逐个验证：刷新后商家/骑手只出现在合理锚点范围，运行 10 秒后派单线数量等于商家数，线路贴近道路层。
- 验证“刷新地图/位置”按钮确实改变点位但保持场景特征。
- 分别验证“刷新位置”和“刷新地图”：刷新位置必须改变骑手/商家点位和候选派单关系；刷新地图必须改变地图视觉/道路/交通变体，但不能被误解为只是换一批骑手和商家。
- 验证地图工具按钮、商家/骑手/线路/策略表点击详情均可用。
- 保存截图与审计 JSON 到 `goal/goal-8/`。

## 回滚方案

仅回退本轮改动文件，优先使用 `git diff` 与 `apply_patch` 精准撤销。不使用 `git reset --hard` 或 `git checkout --`。如外部资源或复杂依赖引入失败，回退到本地程序化地图纹理和 SVG 线路方案。
